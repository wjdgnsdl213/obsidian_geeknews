# AI 산출물 검증 에이전트 (no-man 리뷰어)

## 문제정의
조직에 도입된 LLM(ChatGPT/Copilot 등)은 RLHF로 과잉 정렬되어 사용자에게 동의만 하는 "yes-men"이 되며([[생산적인 개인이 생산적인 기업을 만들지는 않는다]]), [[우리 직장의 LLM 집단 망상]]에서처럼 AI가 만든 보고서/제안서/코드를 그대로 믿고 제출하는 "workslop"이 조직 내에 쌓인다. 산출물의 근거·위험·누락을 짚어주는 별도의 검증 주체가 없다.

## 해결방법
AI 산출물(보고서/제안서/코드/이메일)을 별도 context의 "verifier sub-agent"에 입력하면, [[How to Be a 30x AI Engineer with a Taste]]의 품질 루브릭(정확성/유지보수성/효율성/보안/스타일 등 도메인별 커스터마이즈)을 기준으로 ① 근거가 부족한 주장 ② 누락된 리스크 ③ 검증 안 된 수치를 짚어내는 "no-man" 역할을 수행한다. [[Loop Engineering]]의 maker-checker 구조(작성자와 검증자 분리)를 그대로 구현.

## 사용데이터
- 검증 대상 문서/코드 (사용자 업로드)
- 도메인별 평가 루브릭 (자체 정의, 예: 보안 체크리스트, 데이터 분석 보고서 체크리스트)
- (선택) 과거 검증 이력을 누적하는 메모리([[supermemory]] 연동 가능)

## 핵심기능
- 문서/코드 업로드 → 루브릭 기반 자동 채점 + "이 주장의 근거는?" 형태의 질문 리스트 생성
- "신뢰도 점수"와 함께 수정 제안 제시 (단순 칭찬형 피드백 금지 — 명시적으로 비판적 톤 유지)
- 검증 이력 누적으로 동일 유형 실수 반복 여부 추적

## 차별점
- 시중 AI 도구는 "더 잘 쓰게" 도와주지만, 이 도구는 "AI가 만든 것을 믿지 말고 검증하라"는 반대 포지션 — Institutional AI 트렌드(Bias 축)에 정확히 부합
- 정훈의 보안 배경을 살려, 코드 산출물 검증 시 [[Microsoft의 오픈소스 도구가 해킹되어 AI 개발자들의 비밀번호 탈취에 악용됨]]에서 다룬 공급망 위험 항목도 체크리스트에 포함 가능

## MVP범위
- 도메인 1개(예: 데이터 분석 보고서 또는 코드 PR)에 한정한 루브릭 정의
- Claude API로 "문서 입력 → 루브릭 점수 + 비판적 피드백" 출력하는 CLI/웹 데모
