# How to Be a 30x AI Engineer with a Taste

**원문:** https://pakodas.substack.com/p/how-to-be-a-30x-ai-engineer-with-a-taste

## 요약
AI 시대 엔지니어 가치는 코딩 속도가 아닌 "안목(taste)"

## 핵심 포인트
- 코드 생성이 commodity가 된 시대, 가치는 "무엇을 만들지/어떤 게 충분히 좋은지/언제 멈출지"를 판단하는 안목(taste)으로 이동 — 인식(recognition)·나침반(compass)·비전(vision) 세 형태로 구성됨
- 안목이 작동하는 5개 가치 영역: ① 문제 선정 ② 시스템 아키텍처 ③ 품질 판단 ④ 사용자 공감 ⑤ 커뮤니케이션/스토리텔링 — 각각 구체적 사례(Claude Code, Codex, OpenClaw) 제시
- 90일 안목 훈련 계획(1개월 구조화된 노출/연구, 2개월 비교 분석, 3개월 직접 제작)과 5개 포트폴리오 프로젝트(AI 생성 코드 평가 프레임워크, 온보딩 재설계, 팀 안목 테스트, 48시간 제품 출시, 사고 전환형 블로그 글) 제시

## 왜 중요한가
"AI가 코드를 쓰는 시대, 엔지니어의 가치는 평가·판단 능력"이라는 프레임은 정훈의 면접에서 "AI 시대에 왜 사람이 필요한가"에 대한 핵심 답변 논리가 되며, 제시된 5개 프로젝트(특히 "AI 생성 코드 평가 프레임워크")는 포트폴리오로 즉시 구현 가능한 구체적 스펙을 제공한다.

## 내 관심 분야 적용
- 취업/면접: "AI 시대 엔지니어의 핵심 역량"을 안목(recognition/compass/vision) 프레임으로 설명하고, "30% 직접 작성 70% AI" 같은 구체적 수치 사례 인용
- 공모전: "AI 생성 코드/콘텐츠 품질 평가 자동화 툴" 아이디어 — 정확성·유지보수성·보안·스타일 루브릭을 정의해 점수화하는 서비스
- 포트폴리오: 5개 프로젝트 중 "AI 생성 코드 평가 프레임워크"를 50개 실제 PR에 적용해 루브릭/결과를 공개하는 미니 프로젝트로 구현
- 사업: 소상공인 대상 "AI로 만든 결과물(코드/콘텐츠)의 품질을 평가해주는" 컨설팅 서비스 설계 시 루브릭 구조를 참고

## 원문 스크랩

**핵심 주장**: 2025년 11~12월 Opus 4.5/GPT-5.2/Gemini 3 이후 코드 생성은 commodity화됨. 남은 가치는 "everything else" — 문제 분해, 무엇을 만들지 결정, 테스트가능성/신뢰성/확장성을 위한 아키텍처, 사용자 이해, 충분히 좋은지 판단. 이것이 "taste".

**Taste의 3형태**: ① Recognition(인식) — 두 구현을 보고 어느 게 나은지 설명 전에 느끼는 능력 (예: Codex팀이 Rust를 선택한 이유는 기술적 우월성이 아니라 팀 문화 효과). ② Compass(나침반) — 무엇을 만들지 방향을 감지 (Boris Cherny가 Claude Code의 todo list 기능을 2일간 20개 프로토타입으로 반복한 사례). ③ Vision(비전) — 2년 뒤 무엇이 중요할지 아는 능력 (Peter Steinberger가 "대부분의 코드는 지루한 데이터 변환, 에너지는 시스템 설계에" 라고 한 것).

**5개 가치 영역**: ① 문제 선정(어떤 문제를 풀면 5개 문제가 같이 사라지는가) ② 시스템 아키텍처(모델 출시마다 코드를 삭제하는 Claude Code 철학) ③ 품질 판단(Codex팀의 계층적 코드리뷰 — 핵심 코드만 인간 리뷰, OpenAI 엔지니어 70%는 AI로 작성, 나머지 30%가 품질 판단이 중요한 부분) ④ 사용자 공감(Codex팀이 채택률을 희생하고 샌드박스를 기본값으로 설정) ⑤ 커뮤니케이션(OpenClaw가 Claude Code+Codex 합산보다 더 많은 검색량, AGENTS.md 같은 AI 대상 문서화).

**90일 계획**: 1개월 — 개발자 도구 10개·연구 논문 10개를 "결정"의 관점으로 분석. 2개월 — 매주 두 구현을 비교해 "왜 더 나은가"를 500자로 작성, 매일 "왜 이걸 선택했고 다른 대안은 아닌가" 1문장 기록. 3개월 — 자신의 시스템 재설계, 최고의 글쓰기, 처음부터 시스템 설계 후 모든 결정을 1원칙으로 설명, 자신의 안목을 문서/시스템으로 인코딩.

**5개 포트폴리오 프로젝트**: ① AI 생성 코드 50개 PR에 정확성/유지보수성/효율성/보안/스타일 루브릭 적용 후 결과 공개 ② 좋아하는 오픈소스 프로젝트의 온보딩 5분 재설계 ③ 팀용 "안목 테스트"(구현 방식 10쌍 비교, 의견 불일치 지점 분석) ④ 48시간 제약으로 실제 사용자가 쓸 제품 출시 ⑤ 독자의 사고를 바꾸는 기술 블로그 글 작성.

**커리어 전략**: 속도 경쟁 중단, 제품/비즈니스/디자인 같은 인접 역량 투자, 안목이 보상받는 역할(founding engineer, tech lead, platform engineer, DX engineer, staff+) 선택, 이력서보다 강력한 "안목을 보여주는 공개 포트폴리오" 구축.
