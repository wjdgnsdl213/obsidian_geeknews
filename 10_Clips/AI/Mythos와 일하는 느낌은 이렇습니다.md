# Mythos와 일하는 느낌은 이렇습니다

**원문:** https://news.hada.io/topic?id=30368

## 요약
Mythos급 모델 Fable 5 사용 후기: '마법사'에서 '후원자'로

## 핵심 포인트
- Claude Fable 5는 다단계 사양서를 주면 최대 12시간 자율 작업하며, 작업 중 더 저렴한 Claude Sonnet을 직접 실행해 연구·코딩·검증을 분담(등시선 지도 제작 사례: 2,200건 항공편/철도 시간표/도로 속도 데이터 수집, 적대적 에이전트 그룹으로 상호 검증)
- 19페이지 설계문서를 9.5시간 작업해 "Concord"라는 연구용 데이터 보정 소프트웨어를 완성 — 전문가가 일부 오류·누락을 발견했지만 기존에 없던 수준의 결과물
- HN 반응은 양극화: "검증이 핵심이며 코드 품질·문서·테스트에 대한 정보가 전혀 없다"는 비판 vs "관찰 가능한 동작이 좋으면 코드 모양은 중요하지 않다"는 의견, "9.5시간/주간 한도 16% 소모"라는 비용 문제 지적

## 왜 중요한가
"인간의 역할이 마법사(직접 조종)에서 후원자(의뢰·판정)로 전환"되는 흐름과 "검증이 핵심 병목"이라는 HN의 반박은 정훈이 면접에서 "AI 에이전트 시대 사람의 역할"을 설명할 때 두 가지 관점(낙관/비판)을 균형 있게 제시할 수 있는 소재이며, "적대적 에이전트 그룹으로 상호 검증" 패턴은 [[Loop Engineering]]의 maker-checker 구조와 직접 연결된다.

## 원문 스크랩

**성능**: Fable 5는 최대 12시간 자율 작업, 단일 프롬프트+1회 피드백으로 정교한 사회과학 논문과 모든 단어가 s로 시작하는 10페이지 운율시 생성. Claude Code에서 모호한 프롬프트만으로 플레이 가능한 게임 다수 제작(이미지 생성 불가하므로 모든 아트를 수학 연산으로 구현).

**등시선 지도 사례**: 도시별 이동시간 지도 제작을 위해 Fable이 다수의 저렴한 Sonnet 에이전트를 직접 실행해 TGV~Shinkansen 철도시간표, 국가별 도로속도, 2,200건 이상 항공편 데이터 수집. 원격지(Greenland, Pitcairn Island) 보정에는 "적대적 에이전트 그룹"이 서로 검증하는 워크플로 사용. 짧은 시간에 막대한 토큰 소비.

**Concord 사례**: 인간/AI 응답을 보정하는 연구용 소프트웨어를 19페이지 설계문서 기반 9.5시간 작업으로 완성. 전문가가 일부 오류·누락 발견 후 수정 지시. "연구자들이 수년간 필요로 했으나 수익성이 없어 만들어지지 않던 소프트웨어".

**한계**: Opus 대비 토큰 2배 비용, 보안 관련 미세한 기미에도 가드레일이 작동해 Opus 4.8로 전환되는 일이 빈번, jagged frontier와 "Claudism" 문체 잔존.

**마법사→후원자**: 작업이 과정에서 결과로 이동, 더 이상 조종(steer)이 아닌 의뢰(commission). 모델이 자체 에이전트를 띄워 조사·작성·상호검증까지 완료 — "스튜디오 전체에 의뢰"하는 형태.

**HN 비판**: "검증이 핵심 병목"(코드 품질·테스트·문서 정보 부재), "관찰 가능한 동작이 좋으면 코드 모양은 안 중요"는 반박도 있음. 비용 측면에서 40분 코드리뷰로 주간 한도 16% 소모 사례, "투자 대비 수익이 핵심 질문"이라는 지적. "소프트웨어 엔지니어가 남은 버그를 다듬을 것"이라는 가정은 비현실적이라는 비판도 다수.
